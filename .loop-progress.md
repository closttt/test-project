### POST-LOOP HOTFIX — real crash reported by the user after this session, root-caused and fixed
User reported "что-то пошло не так" (the `ErrorBoundary` fallback) on startup, with the exact React
error visible: `Element type is invalid: expected a string... but got: undefined. Check the render
method of MobileNav.` **Root cause**: `Sidebar.tsx` and `MobileNav.tsx` each keep their OWN separate
`ICONS: Record<string, Icon>` map keyed by nav item id — a duplication that existed before this session,
not introduced by it. When W3 added `{ id: "clients", ... }` to `NAV_ITEMS` (`navConfig.ts`) to wire up
the new Клиенты page, neither `ICONS` map got a matching `clients` entry. `loadNavOrder()` auto-appends
any new nav id not yet in the user's saved order, so "clients" landed in both the full Sidebar list and
(being within the top 5) the MobileNav bottom bar — `ICONS["clients"]` resolved to `undefined`, and
`<Icon />` with `Icon=undefined` is exactly this crash. Crashed on literally every page load since both
components mount unconditionally in `AppLayout`. **Not caught by `tsc`/`vitest` during the original
session** — `Record<string, Icon>` lookups don't fail to typecheck on a missing key, and no test had
ever actually rendered `Sidebar`/`MobileNav` before this incident.
- **Fix**: added `clients: Users` to both `ICONS` maps (`Users` — same icon already used for clients in
  `CommandPalette.tsx`/`EmptyState.tsx`).
- **Regression test added**: `src/components/layout/__tests__/nav.test.tsx` — renders `Sidebar`/
  `MobileNav` with real providers and asserts every `NAV_ITEMS` label appears without throwing.
  **Verified the test actually catches the bug**: temporarily reverted the fix, confirmed the test fails
  with the identical error class, then restored the fix and confirmed green — not just written on faith.
- 82 → 85 tests. tsc/vitest/build all clean after the fix.
- **Lesson for future nav-item additions**: `Sidebar.tsx` and `MobileNav.tsx` have independent icon maps
  that must both be updated whenever `NAV_ITEMS` (`navConfig.ts`) gains a new entry. This regression test
  now catches it automatically, but the duplication itself is a standing footgun — consolidating the two
  `ICONS` maps into one shared export would remove the footgun at its root, not just add a test around
  it. Not done in this hotfix (kept the fix minimal and targeted); worth a future small `DEEPEN` pass.
