# Информационная архитектура MVP

Синтез из `brief.md` + `research.md`, решён интуитивно (без дополнительного раунда вопросов) — открытые
пункты брифа закрыты ниже с пометкой «почему».

## Главное действие пользователя

Открыть приложение → увидеть, что горit сегодня (задачи + встречи + клиенты без движения) → либо
разобрать список, либо быстро добавить новую задачу одной кнопкой с любого экрана.

## Модель данных

- **Client** — `id, name, company?, email?, phone?, status(active|negotiation|archived), revenue,
  expectedPayment, lastActivityAt, notes?, createdAt`
- **Project** — `id, name, clientId?(nullable), status(active|done|archived), createdAt`
  — связь с клиентом **опциональна**: закрывает открытый вопрос брифа, разрешает внутренние проекты
  без клиента.
- **Task** — `id, title, projectId?(nullable), done, dueDate?, subtasks: {id,title,done}[], createdAt`
  — связь с проектом **опциональна**: закрывает второй открытый вопрос, разрешает свободные задачи
  вне проекта (Inbox-паттерн из скрина 2). Вложенность — ровно 2 уровня, без под-подзадач.
- **Note** — `id, title, body, linkedClientId?, linkedProjectId?, createdAt` — связи необязательные.
- **Meeting** — `id, title, clientId?, date, time` — обслуживает календарь и дашборд.

**Правило «клиент в зоне риска»** (было открытым пунктом брифа, закрыто по рекомендации research.md):
`daysSinceLastActivity = today − lastActivityAt`. `< 14` → норма, `14–29` → «внимание» (жёлтый),
`≥ 30` → «риск» (красный). Порог мягче типичного CRM-дефолта (30–90 дней для холодных лидов), т.к.
это уже действующие клиенты, а не воронка продаж.

## Карта экранов

| Экран | Назначение | Сущности | Главное действие |
|---|---|---|---|
| Дашборд (`/`) | Сводка на сегодня | Task, Meeting, Client | Отметить задачу / открыть клиента в риске |
| Клиенты (`/clients`) | Реестр клиентов | Client | Добавить / открыть клиента |
| Проекты (`/projects`) | Проекты по клиентам | Project, Client | Добавить проект |
| Задачи (`/tasks`) | Все задачи + чек-листы | Task, Project | Добавить задачу / отметить подзадачу |
| Заметки (`/notes`) | Свободные заметки | Note | Добавить заметку |
| Календарь (`/calendar`) | Ближайшие встречи | Meeting, Client | Добавить встречу |

## Навигация верхнего уровня

Порядок: **Дашборд → Клиенты → Проекты → Задачи → Заметки → Календарь** — приоритет по частоте
использования в JTBD брифа (деньги/клиенты сразу после сводки, задачи в середине как рабочая лошадка,
заметки и календарь — вспомогательные).

**Глобальный быстрый ввод** — кнопка «+ Добавить» в шапке на всех экранах, открывает диалог создания
задачи (самый частый акт захвата). Решает найденное в research.md напряжение «дашборд-как-дом vs
Inbox-паттерн»: дашборд остаётся местом обзора, а быстрый ввод не привязан к конкретному экрану.

## User flow главного сценария

1. Открыть приложение → дашборд.
2. Увидеть 3 блока: задачи на сегодня, ближайшие встречи, клиенты в зоне риска.
3. Отметить выполненные задачи прямо на дашборде (чекбокс).
4. Увидеть клиента в риске → перейти в карточку → обновить статус/заметку → `lastActivityAt` обновляется.
5. Если появилась новая задача — нажать «+ Добавить» из шапки, не покидая текущий экран.

## MVP-скоуп vs отложено

**В MVP:** всё вышеперечисленное, локальное хранение (localStorage), тёмная тема без переключателя.

**Отложено** (из anti-scope брифа): графики/аналитика, канбан сделок, тайм-трекинг/ставки,
уведомления/интеграции/вложения, мультипользовательский режим, бэкенд (Supabase — следующая волна).
