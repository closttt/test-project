# v3.0 «Фокус» — roadmap и дизайн-спека

**Дата:** 2026-07-13
**База:** v2.1 (см. `version-2.1.md`).
**Триггер:** запрос — проанализировать методологии тайм-менеджмента, упростить интерфейс не теряя пользы, и параллельно закрыть большой пакет точечных правок (Agentation-фидбэк по /, /projects, /tasks, /analytics, /notes, /achievements).
**Формат прохода:** без уточняющих вопросов по ходу (явное указание пользователя) — решения по неоднозначным местам принимались автономно и зафиксированы здесь.

---

## 1. Исследование: методологии тайм-менеджмента → выводы для интерфейса

Разобраны: GTD, матрица Эйзенхауэра, Pomodoro, Deep Work, Essentialism, Atomic Habits, Eat That Frog,
Building a Second Brain/PARA, Four Thousand Weeks, The One Thing, метод Иви Ли, закон Паркинсона,
тайм-блокинг.

**Синтез — 10 принципов упрощения** (важно: не только «добавить», но и «убрать/приглушить»):

1. Свернуть виджет-дашборд в опциональный вид — по умолчанию один ранжированный список «Сегодня», а не конкурирующие панели.
2. Ограничить дневной список 3–6 пунктами (Essentialism/метод Иви Ли) — остальное само уезжает на завтра, а не тонет в скролле.
3. Аналитика — по запросу, а не постоянно на глазах (числа сами по себе отвлекают — Four Thousand Weeks).
4. Уменьшить визуальный вес геймификации — стрики оставить, но не превращать в лидерборд.
5. Канбан — вторичен по отношению к календарю (тайм-блокинг уже упорядочивает работу).
6. Один путь захвата задачи (GTD) — один global quick-add → один инбокс, а не 4 разных «+».
7. Заметки организуются по actionability (PARA), а не по ручным тегам — архив одним кликом.
8. Один сигнал приоритета, не четыре — было "звёздочка + флаг + P1/P2/P3 + цвет колонки"; сведено к одному цветному индикатору.
9. Настоящий режим фокуса — помодоро должен приглушать остальной интерфейс, а не просто тикать рядом.
10. Разрешить приложению говорить «нет» — предупреждение о перегрузке вместо бесконечно растущего бэклога.

**Что это значило для факта:** пункты 8 (приоритет одним цветным индикатором) и 6 (три точки входа →
единый bottom-dock) реализованы в этом проходе напрямую. Остальные — направление для v3.x, см. §5.

## 2. UX-аудит текущей реализации (до правок)

Ключевые находки (полный список смотрел по коду, не «на глаз»):

- Дашборд дублировал «сегодняшний список» рядом с отдельной страницей `/focus` — та же информация в двух местах.
- Три разных «+»-кнопки с одинаковым визуальным весом (глобальный «Добавить» + свои у Projects/Notes/Calendar) путают, какая что делает.
- `Dashboard.tsx` создавал задачу в обход `QuickAddDialog`/NLP-парсера и без тоста — то же действие, два разных набора возможностей.
- Скрытие виджета дашборда не имело undo, хотя весь остальной интерфейс — да (несогласованность).
- Добавление встречи с дашборда использовало нативный `window.prompt` вместо своего диалога.
- Цифровые хоткеи 1–8 (списки на /tasks) не подписаны на самих пилюлях — нужно считать по счёту.
- Приоритет была голым текстом без чипа/hover-состояния — не читалось как интерактивный элемент.
- `AreaChart`: фиксированный `viewBox="0 0 100 100"` + `preserveAspectRatio="none"` → нелинейное растяжение линии графика при любом отклонении от квадрата (то есть почти всегда).
- Ресайз-хендл дашборда рисковал обрезаться `overflow-hidden` на скруглённом углу контейнера — тот же баг был явно описан ещё в спеке v2.0 («не клипать угловую ручку overflow'ом»), похоже недокрутили или регресснуло.

## 3. Что сделано в этом проходе

### Дашборд (`/`)
- `AreaChart` переписан — `viewBox` теперь совпадает с реальным размером контейнера (ResizeObserver), растяжения больше нет физически (не «подправили визуально», а убрали причину).
- Приоритет — цветной чип (как тег), перенесён вправо от названия задачи.
- Убраны: «≈ время» в шапке списка «на сегодня», «потрачено/оценка» и кнопка таймера в строке задачи (шум).
- Быстрая заметка сохраняется по ⌘/Ctrl+Enter.
- Время встречи — с иконкой; добавление встречи — нормальный диалог (дата/время) вместо `window.prompt`.
- Клик по «Открыто задач» → попап со списком всех открытых задач (чекбокс/приоритет/срок).
- Клик по «Закрыто за неделю» → попап с разбивкой по приоритетам.
- Ресайз-хендл больше не может быть обрезан скруглённым углом контейнера в режиме редактирования (убран `overflow-hidden` с внешней обёртки виджета на время editing).
- **Не удалось проверить вживую**: браузер Playwright был занят другой сессией весь проход — жалоба «не могу тянуть/ресайзить» не воспроизведена вручную, применён наиболее вероятный fix по коду. Стоит перепроверить глазами.

### Bottom dock + шапка
- Новый компонент `BottomDock` — поиск (⌘K) / добавить задачу / помодоро одним рядом внизу по центру экрана.
- Из шапки (`AppShell`) убраны Search/Добавить — остались только Инсайты и Story (Changelog).
- Помодоро: клик по иконке в доке, когда сессия не идёт, открывает выбор задачи (или «без задачи») — раньше FAB стартовал без задачи всегда.

### Задачи (`/tasks`)
- Приоритет — тот же цветной чип, в списке и в диалоге.
- Теги ограничены тремя: клиент / личное / работа (чипы вместо свободного текста и в фильтре, и в диалоге редактирования).
- Сортировка списка: вручную / по дате / по приоритету (сегмент рядом с поиском).
- Канбан — добавлен четвёртый режим «Доска»: свои колонки (добавить/переименовать двойным кликом/удалить с переносом задач/перетащить для смены порядка), поверх уже существующих «Приоритет/Проект/Статус».
- Диалог задачи: приоритет — 4 цветных кружка вместо `P1/P2/P3`-текста; вложения — прикрепление файлов (хранятся в IndexedDB, не в localStorage — метаданные в `Task.attachments`).
- Глобальный undo (см. ниже) подключён к удалению/архивации/массовым действиям.

### Глобальный Ctrl+Z
- `src/lib/undoStack.ts` — общий стек одноразовых восстановлений. Существующие тосты «Вернуть» и Ctrl+Z используют ОДНУ и ту же обёрнутую функцию — что сработает первым, то и «съедает» запись (без двойного восстановления).
- Подключено: удаление/архивация задачи и массовые операции, удаление/архивация заметки, скрытие виджета дашборда, удаление колонки канбана.
- Не подключено (сознательно, вне бюджета этого прохода): перетаскивание в календаре, полная история на 50+ шагов везде — см. backlog.

### Проекты (`/projects`)
- Карточка проекта показывает превью открытых задач (до 4, `+N ещё`) с приоритетом и сроком — не весь бэклог целиком (см. §4 — почему).

### Заметки (`/notes`)
- Карточка: заголовок + первая строка текста (обычным текстом, не рендер Markdown) + теги + проект. Полный текст, дата — убраны с карточки (открываются по клику на редактирование).

### Аналитика (`/analytics`)
- `AreaChart`-фикс наследуется автоматически (общий компонент).
- KPI «Закрыто за N дн.» и график «Открытые по приоритету» теперь кликабельны — открывают те же попапы, что на дашборде (список задач).

### Достижения (`/achievements`) — фоновый агент
- Добавлен блок «Рекорды»: лучший стрик, максимум задач/день, максимум фокуса/день, максимум помодоро/день, самая длинная сессия.
- +15 новых достижений (было 15, стало 30): вехи (250/500 задач, 100/200 помодоро), «сова»/«жаворонок» по времени завершения, «перфекционист» (0 просрочек при 3+ со сроком), «возвращение» после паузы в стрике, ударный день, уровни 15/20 (титулы расширены до 20).
- Level-up: тост теперь называет титул уровня, плюс лёгкий scale/glow-пульс карточки поверх существующего конфетти.

### Шорткаты
- В `ShortcutsDialog` (клавиша `?`) добавлен компактный блок «быстрое добавление» — синтаксис quick-add (даты словами, `#тег`, `!важно`), который раньше был только в подсказке самого диалога быстрого добавления.

## 4. Про раздел «Проекты» — отдельный большой фокус (СЛЕДУЮЩИЙ проход, не в этом)

Пользователь отдельно уточнил по ходу: **проекты — по сути более важная сущность, чем задачи** — это
«история», которую ведут вплоть до 100–200 задач внутри, с картинкой, заметками/комментариями по
проекту, и подразделами-заголовками (частично уже есть — `Project.sections`/`Task.section`, поставлено
в 2.1). Явно попросил проработать это **детально, отдельным проходом** после текущих правок — поэтому
здесь фиксируется дизайн-направление, а не код.

**Что уже сделано сейчас (совместимо, не мешает будущему проходу):** превью задач на карточке проекта в
списке (см. §3) — специально ограничено 4 штуками именно из-за этого требования (100–200 задач в
проекте нельзя вывалить на карточку в гриде).

**План для отдельного прохода (детальная проработка):**
- **Изображение проекта** — обложка карточки. Технически: тот же механизм, что уже завезён для вложений
  задач (`src/lib/attachments.ts`, IndexedDB) — добавить `Project.coverAttachmentId?`.
- **Заметки/комментарии на уровне проекта** — не только заметки, привязанные через `linkedProjectId`
  (уже есть), а именно лента комментариев/апдейтов внутри страницы проекта: «что нужно внести»,
  «что сделать под задачей», хронологический журнал. По структуре — как `TaskComment`, но на `Project`.
- **Подразделы (sections) — усилить.** Сейчас это плоский список имён + `Task.section`. При 100–200
  задачах нужны: сворачиваемые секции, счётчик/прогресс на секцию, порядок секций управляемый (drag),
  возможность видеть секцию как мини-канбан (переиспользовать новый механизм колонок из `lib/kanban.ts`,
  но на уровне секции проекта, а не глобальном виде задач).
- **Приоритеты и навигация при большом объёме** — фильтры/сортировка внутри `ProjectDetail.tsx` по
  аналогии с тем, что теперь есть на `/tasks` (по дате/приоритету), плюс, возможно, сохранённые виды на
  уровне проекта.

Это осознанно НЕ реализовано в этом проходе (риск: без отдельного brainstorming/spec-цикла легко
«угадать неправильно» такую крупную сущность) — вынесено в roadmap как первый по приоритету пункт «Можно
сейчас» (см. `lib/roadmap.ts`).

## 5. База знаний через Telegram — техническое решение

**Обновлено:** пользователь подтвердил стек — **Hermes Agent** (не общий "какой-то Hermes", а конкретный
продукт NousResearch, `hermes-agent.nousresearch.com`) через **Hermes Desktop** (нативное GUI,
вышло 2026-06-02, v0.15.2 public preview, macOS/Windows/Linux) + **Supabase** как общий бэкенд.
Проверено веб-поиском (открыт после моего knowledge cutoff в январе 2026):

- Hermes Agent — open-source автономный агент (MIT, 180k+ звёзд на GitHub за 4 месяца), **model-agnostic**:
  можно подключить OpenAI, Anthropic, OpenRouter, локальные модели через Ollama/llama.cpp, или Nous Portal
  — то есть сам Hermes Agent НЕ модель, а рантайм/оболочка поверх любой модели на выбор. Это снимает вопрос
  «какую нейронку покупать» — можно начать с бесплатной локальной (Ollama) и повысить позже без смены агента.
  Хранит персистентную память, сам строит переиспользуемые «скиллы» из задач.
  ([nousresearch.com](https://hermes-agent.nousresearch.com/), [github.com/NousResearch/hermes-agent](https://github.com/nousresearch/hermes-agent))
- У Hermes Agent уже есть встроенный **gateway на 24 платформы, включая Telegram, Discord, Slack, WhatsApp** —
  то есть отдельный кастомный Telegram-бот писать не нужно, канал уже есть из коробки.
- Desktop-приложение — нативный GUI поверх того же агента, поддерживает локальные OpenAI-совместимые
  эндпоинты (LM Studio, Ollama, vLLM, llama.cpp) для выбора модели.
  ([hermes-ai.net/desktop](https://hermes-ai.net/desktop/))

**Итоговая архитектура:**
1. Hermes Agent запущен где-то персистентно (свой ПК, если ок работать только пока он включён, либо
   небольшой VPS для «работает всегда») — подключён к Telegram-гейтвею (токен бота через @BotFather).
2. Модель для агента — на старте Ollama (локально, $0) или любой уже оплаченный ключ (OpenAI/Anthropic/
   OpenRouter), если нужно качество получше — переключается в настройках агента без смены архитектуры.
3. Агенту даётся кастомный скилл/тул: получил пересланный пост → достал заголовок/описание/картинку
   (картинка — напрямую из Telegram через Bot API `getFile`, без LLM) → записал строку в Supabase
   (`service_role` ключ, никогда не попадает в браузер).
4. **Уже готово в этом проходе (код в CRM):**
   - `supabase/knowledge_cards.sql` — схема таблицы + RLS-политика (анон-ключ — только чтение).
   - `.env.example` → `.env.local` (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`).
   - `src/lib/supabase.ts` — тонкий REST-клиент (без SDK-пакета, тот же принцип «прямой fetch», что и
     в дизайне AI-ассистента v2.0), `src/types.ts` → `KnowledgeCard`.
   - Страница `/knowledge` (пункт в сайдбаре и ⌘K) — грид карточек (картинка/заголовок/описание/теги/
     ссылка на источник), с аккуратным экраном «Supabase не настроен» вместо краша, пока переменные не заданы.
5. **Что дальше нужно от пользователя** (не автоматизируется мной): создать проект на supabase.com,
   выполнить SQL-схему, вписать переменные окружения, установить Hermes Desktop и настроить его Telegram-
   гейтвей + скилл записи в Supabase (это конфигурация уже установленного Hermes Agent, не код CRM).

## 5.1 Старая версия раздела (для истории — что было неизвестно на момент первого прохода)

Запрос: переслать пост из Telegram боту → бот через бесплатный API оформляет карточку (заголовок,
описание, картинка из поста) → карточка сама появляется в базе знаний приложения.

**Ключевое архитектурное ограничение:** сегодня это чисто фронтенд-приложение (localStorage в браузере,
без бэкенда). Telegram-бот физически не может писать в `localStorage` чужой открытой вкладки — значит,
эта фича **зависит от уже существующего пункта бэклога «облако-синк» (Supabase)**, помеченного как
«Нужно решение». Без общего облачного хранилища бот и приложение — два разных мира.

**Предлагаемый пайплайн (когда облако-синк будет решён):**
1. Telegram Bot API (бесплатно) — простой бот на вебхуке (Cloudflare Worker / Vercel Function / малый
   Node-процесс) принимает пересланное сообщение (текст + фото).
2. Вызов LLM для структурирования: заголовок + описание + теги из сырого текста поста.
3. Картинка берётся напрямую из Telegram-сообщения (Bot API отдаёт `file_id` → `getFile` → URL) — LLM
   для этого не нужен, только для текста.
4. Результат пишется в облачную БД (Supabase) как новая карточка «база знаний»; приложение подтягивает
   её через тот же синк-канал, что и остальные данные.
5. В UI — новая сущность `KnowledgeCard` (заголовок, описание, картинка, источник-ссылка, теги),
   отдельная страница/раздел, не смешивать с обычными заметками (разное назначение: заметки — свои
   мысли, база знаний — сохранённый чужой контент).

**Какую нейронку подключать — прямой ответ на вопрос:**
- **Gemini (Google AI Studio) — рекомендация по умолчанию.** У него по-настоящему бесплатный тариф
  (Gemini 2.0/2.5 Flash) с щедрыми лимитами, для «пришёл 1 пост → вынь заголовок/описание» этого более
  чем достаточно, и он отлично работает со структурированным выводом (JSON). Прямой `fetch`, ключ не
  нужно платить.
- **Hermes (Nous Research)** — это открытая модель, не отдельный хостинг-сервис. Доступ к ней бесплатно
  (с лимитами) — через **OpenRouter**, который агрегирует много моделей. Если у пользователя в виду
  конкретный продукт «Hermes Agent» — не удалось однозначно определить, что это; если пришлёте
  ссылку/источник, распишу предметно, а не гадаю.
- **Локально (Ollama + Llama 3.1 / Qwen2.5 / Hermes 3 GGUF)** — $0 навсегда, данные не покидают
  компьютер, но качество извлечения слабее облачного и нужен постоянно запущенный Ollama рядом с ботом
  (или на том же сервере, где живёт вебхук).
- **Платный Claude/OpenAI** — не обязателен для этой конкретной задачи (разбор одного поста — лёгкая
  задача, бесплатного тарифа Gemini хватает с большим запасом). Платный смысл появляется, если объём
  вырастет на порядки или понадобится более тонкое понимание контекста/логика уровня агента — не этот
  случай. Рекомендация: начать с Gemini free tier, не платить, пока не станет тесно по лимитам.

Этот раздел — план, реализация не входит в этот проход (зависит от cloud-sync, который сам помечен «Нужно решение»).

## 6. Backlog после этого прохода

**Правило (новое в 3.0):** любой пункт, который появляется в спеке/roadmap-документе, ВСЕГДА должен
попасть в `DEFAULT_PLANNED` (`src/lib/roadmap.ts`), а не только в markdown-документ — иначе он не виден
внутри продукта. Раньше новые пункты `DEFAULT_PLANNED` не доходили до пользователей с уже сохранённым
`crm-roadmap-v1` (сид применялся только при первой загрузке). Починено: `loadPlanned()` теперь мёржит
новые пункты в существующий сохранённый роадмап автоматически, ровно один раз на пункт (трекинг
«показано» — `crm-roadmap-seen-v1`), не трогая то, что пользователь уже удалил или отредактировал.
