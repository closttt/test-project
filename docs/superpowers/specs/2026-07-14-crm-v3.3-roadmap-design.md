# v3.3 «Внутрянка» — roadmap и дизайн-спека

**Дата:** 2026-07-14
**База:** v3.2 (см. `version-3.2.md`).
**Триггер:** запрос — большой пакет доработок внутренней эргономики и микро-взаимодействий (не новые
разделы, а полировка существующих), плюс research по конкурентам и аудит текущей реализации перед
правками. Одобрено пользователем целиком, одним заходом («Продолжай всю работу исходя из нашего ресерча
и roadmap»).

---

## 1. Research: конкурентные паттерны (полные источники — в транскрипте ресёрч-агента)

- **Инлайн-редактирование задачи** — TickTick: тривиальные поля (тег/дата/приоритет) редактируются прямо
  на строке (drag на смарт-лист, правый клик, шорткоды); тяжёлые поля (подзадачи/вложения) — в отдельной
  **не-модальной** боковой панели, страница не блокируется. Todoist: quick-add с NLP-парсингом,
  конфигурируемая панель иконок. Пойманный антипаттерн (Lullabot, Fluid Project): hover-only edit-аффорданс
  не обнаруживается пользователями — нужен видимый, не только hover, сигнификатор.
- **Карточки проектов** — Linear разделяет **Priority** (ручной) и **Health** (производный) — два разных
  компактных индикатора, не один. Notion gallery cards — не больше 2-3 свойств на карточке. Пойманный
  антипаттерн (ClickUp): нагромождение бейджей на карточке — фикс всегда «сжать», не «добавить ещё».
- **Правая панель фильтров** — Notion: одна общая UI-панель, но per-view сохранённое состояние фильтра.
  Паттерн push (раздвигает контент) предпочтительнее overlay для страниц с частым переключением фильтров.
  Активные фильтры — чипами с крестиком у триггера. Пойманный антипаттерн (Smashing Magazine): «замороженные»
  фильтры — счётчики не обновляются вживую, опции спрятаны за вложенными аккордеонами.
- **Hover-тултипы графиков** — GitHub contribution graph раньше использовал нативный `title` (Adrian
  Roselli задокументировал реальную цену: нет тача, нет доступности со скринридером); в 2023 GitHub сам
  заменил на настоящий позиционируемый tooltip-компонент. Лёгкий React-паттерн без библиотек: один общий
  `useState`-слой тултипа на `onMouseEnter/Leave/Focus/Blur`, а не тултип на каждую точку.

## 2. Аудит текущей реализации (по коду, не «на глаз»)

- `TaskEditDialog.tsx` — 500-строчный модальный диалог на ЛЮБОЕ редактирование, но **3 из 4 нужных полей
  уже инлайн** на строке задачи (`Tasks.tsx`): двойной клик по названию (переименование), клик по
  `PriorityFlag` (цикл приоритета), клик по дате (дропдаун). Единственное отсутствующее инлайн-поле — теги.
  Реальный источник трения — не отсутствие полей, а то, что клик по ЛЮБОМУ месту строки (`onClick` на всём
  теле строки, `Tasks.tsx:343-348`) всё равно открывает диалог поверх уже существующих инлайн-контролов.
- `Project` (types.ts:85-98) уже имеет `coverAttachmentId` и `comments?: ProjectComment[]` (сделано в 3.1)
  — но **не имеет** `priority` и никакого поля времени. Задача (`Task`) уже имеет `estimateMin` — можно
  агрегировать по проекту, не заводя дублирующее ручное поле.
- Фильтры/сортировка/поиск — на КАЖДОЙ странице (`Tasks.tsx`, `Projects.tsx`, `Knowledge.tsx`, `Notes.tsx`)
  свой инлайн-бар, разной формы. Правой панели нигде в приложении нет — есть только два fixed-drawer
  оверлея (`InsightsPanel`, `AiAssistant`), не для фильтров.
- Из графиков (`AreaChart`, `BarChart`, `DonutGauge`, `HBarList`) — **ни один** не имеет hover/tooltip.
  `ContributionGrid` — единственный с интерактивностью, но это нативный `title` (устаревший паттерн, см. §1).
- `KnowledgeCard` (types.ts:170-178) — нет `priority`; `src/lib/supabase.ts` имеет только `fetchKnowledgeCards`
  и `deleteKnowledgeCard`, нет `update`; RLS в `supabase/knowledge_cards.sql` разрешает анон-ключу только
  SELECT/DELETE, нет UPDATE-политики.
- `InsightsPanel.tsx` + `lib/insights.ts` — был построен в 2.1 как бесплатная замена платному AI-ассистенту
  (без LLM, только правила по данным). Точки входа: хоткей `i`, кнопка в шапке, пункт ⌘K, строка в
  `ShortcutsDialog`, и ритуальный баннер дашборда (`openInsight("plan"/"week")`).
- «Неделя впереди» — виджет дашборда (`dashboardLayout.ts` `WIDGETS`, `Dashboard.tsx` `case "week-ahead"`),
  уже поддерживает hide/show через общий механизм редактирования дашборда — но это не то же самое, что
  полное удаление из реестра, которое просит пользователь.

## 3. Решения этого прохода

### Убрано
- **Insights** — `InsightsPanel.tsx`, `lib/insights.ts` и все точки входа удалены целиком: хоткей `i`,
  кнопка в `AppShell`, пункт ⌘K, строка в `ShortcutsDialog`, ритуальный баннер дашборда и `useUI` поля
  `aiOpen`/`insightRequest`/`openInsight`.
- **«Неделя впереди»** — удалена из `WIDGETS` (не просто скрыта) и из `Dashboard.tsx`, вместе с `DEFAULT_LAYOUT`.

### Pomodoro — быстрые настройки
Значок-шестерёнка в `BottomDock` (доступна вне сессии) и на `PomodoroBar` (доступна во время фокуса) —
открывает `DropdownMenu` с теми же 4 полями + автостарт, что в Settings, без перехода со страницы.

### Инлайн-редактирование задачи
Клик по телу строки больше не открывает диалог (диалог — только через значок-карандаш / правый клик /
клик по названию в канбане). Добавлен инлайн-пикер тегов на строке (тот же `DropdownMenu`-паттерн, что у
даты). Большой диалог остаётся для тяжёлых полей (подзадачи/вложения/зависимости/ссылки/комментарии/
оценка/повтор/напоминание).

### Проекты — приоритет и время
`Project.priority?: Priority` — тот же `PriorityFlag`, клик-цикл прямо на карточке, без диалога. Время —
**автосумма** `estimateMin` открытых задач проекта (не ручное поле — не дублировать источник правды),
показана рядом с прогресс-баром.

### Фильтрация → правая панель
Общий компонент `FiltersRail` — сворачиваемая правая панель, раздвигает контент (push, не overlay),
закрыта по умолчанию. Поиск остаётся инлайн вверху страницы (как в Notion/Linear — поиск на виду,
фильтры/сортировка/теги/статус уходят в панель). Подключена на `Tasks.tsx`, `Projects.tsx`, `Knowledge.tsx`,
`Notes.tsx`.

### Аналитика/heatmap — hover-тултипы
`useChartTooltip()` + один абсолютно спозиционированный слой на React-state (без новых npm-пакетов).
Подключён к `AreaChart`, `BarChart`, `DonutGauge`, `HBarList`; `ContributionGrid` мигрирован с нативного
`title` на тот же слой (заодно фикс доступности — см. §1).

### База знаний — приоритет по клику
`KnowledgeCard.priority?: Priority` (новая идемпотентная миграция в `knowledge_cards.sql`, RLS-политика
на UPDATE по тому же принципу уступки «личный инструмент», что уже есть у DELETE), `updateKnowledgeCard()`
в `lib/supabase.ts`, `PriorityFlag` прямо на карточке.

## 4. База знаний через Telegram — диагноз (НЕ фиксится в этом проходе)

Разрыв (фото и ссылка не долетают до Supabase, хотя тег долетает) — не баг CRM. Схема (`image_url`,
`source_url`, бакет `knowledge-images`) готова с прошлого прохода. Причина — в кастомном скилле **Hermes
Agent** (Hermes Desktop, вне этого репозитория): судя по симптому, тул записи в Supabase не вызывает
Telegram Bot API `getFile` для фото и не сохраняет исходную ссылку поста перед записью строки через
`service_role`. Рекомендация — сделать `image_url`/`source_url` **обязательными** полями JSON-схемы тула,
которым скилл пишет в базу. Это конфиг Hermes Desktop, не код CRM — держится в памяти как отдельный пункт
для отдельного разговора с пользователем.
