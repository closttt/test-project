## Data layer & persistence

### Fixed
- `src/lib/storage.ts` `migrate()`'s trash-purge `kept()`: a malformed/unparseable `archivedAt` produced `NaN >= cutoff` → `false`, silently treated as "definitely expired" and purged. Now `Number.isNaN(t)` explicitly returns "keep," same as no date at all. Covered by `storage.test.ts`.
- `src/lib/storage.ts` `loadData()`: a `migrate()` throw (malformed persisted record) used to return `null` silently, and the very next `saveData` effect would overwrite the real corrupted-but-recoverable blob with seed data — permanent loss. Now the raw string is backed up to `crm-taskmanager-data-v1-corrupt-backup` before falling back, and the failure is logged. Covered by `storage.test.ts`.
- `src/lib/storage.ts` `saveData()`: now wrapped in try/catch and returns a success boolean instead of throwing from the commit-phase `useEffect` in `DataProvider.tsx` (a `QuotaExceededError` used to crash the whole app to the `ErrorBoundary` fallback). Covered by `storage.test.ts`.
- `src/store/DataProvider.tsx` `deleteProject`: now clears `task.projectId`/`task.section` and `note.linkedProjectId` for anything that referenced the deleted project (mirrors `deleteProjectSection`'s existing reparenting).
- `src/store/DataProvider.tsx` `deleteTask`: now strips the deleted id out of every other task's `blockedBy` array, so a task can't stay "blocked" forever by a dependency that no longer exists.
- `src/store/DataProvider.tsx` `deleteClient`: now clears `project.clientId`/`meeting.clientId` for anything that referenced the deleted client. (Note: the research agent's report claimed a `Task.clientId` field exists — it doesn't; `tsc` caught this immediately. Only `Project` and `Meeting` carry `clientId`.)
- `src/lib/undoStack.ts`: added `clearUndo()`, wired into `DataProvider.tsx`'s `replaceAll` (JSON import/restore) — a pending undo entry from before a wholesale data replacement can no longer resurrect an object into a dataset it no longer belongs to.
- `src/lib/meetingSync.ts` `drainIncomingMeetings`: hoisted the in-flight guard from the two separate call sites (`MeetingSyncEngine`'s 60s poll, `SettingsPage`'s manual "Синхронизировать сейчас") into one module-level promise, so a poll tick racing a manual click can no longer double-import the same invite.

### Deferred
- **Attachment blob cleanup on delete** (`deleteTask`/`deleteProject` don't free `coverAttachmentId`/`attachments`/`photos`/`files` blobs in IndexedDB): deliberately NOT fixed this pass. Both deletes are undoable (`pushUndo` + toast), and eagerly deleting the IndexedDB blob at the moment of `deleteTask`/`deleteProject` would break undo — the record would come back on "Вернуть" but its cover image/attachments would already be gone. The right fix is to hook blob cleanup into the undo-expiry path (or the `trashPurgeDays` purge, which already represents a point past any undo window) instead of the delete action itself — that's a small new mechanism, not a one-line fix, so it's deferred rather than rushed.
- **Debounce on the full-blob `saveData` write**: `Reorder.Group onReorder` (Tasks.tsx, Notes.tsx) fires continuously mid-drag, driving repeated full-`JSON.stringify` writes. The try/catch crash-prevention fix landed this pass; the debounce itself (300–500ms trailing, flush on unmount/`visibilitychange`) did not, to keep this pass's scope to correctness rather than also introducing new timer/cleanup logic.
- **Cross-tab conflict detection** (`storage` event listener): confirmed still absent. Only worth building if the user actually runs multiple tabs/instances regularly — flagging, not fixing speculatively.

## Security & Supabase/backend

### Deferred (explicitly — these need your decision, not a silent patch)
- **`push_subscriptions`/`push_reminders` RLS is `for all using(true) with check(true))`** — broader than the project's own stated single-owner trade-off (which the SQL comments only accept for `knowledge_cards`/`incoming_meetings`, and only read+delete[+update], never insert). Right now the anon key — which ships in the public JS bundle — can read/delete every device's push subscription, and insert/update/delete arbitrary rows in `push_reminders` (including setting `url` to an attacker's domain, which `public/sw.js`'s `notificationclick` then navigates to with no origin allow-list on click). **Not touched this pass** — this is a live Supabase policy change with real deploy implications; AUDIT.md's own guardrail says not to silently fix this. Recommended fix, for your review: drop `select` from `push_subscriptions` (browser only ever writes its own row) and narrow `push_reminders` similarly.
- **`send-reminders` edge function has no shared-secret check** (`--no-verify-jwt`, handler takes no request param at all) — the function URL is trivially derivable from the public `VITE_SUPABASE_URL`, so anyone can invoke it directly. Coupled to the item above (compounds it — a stranger could write a row via the open RLS, then force-fire the function immediately instead of waiting for the cron). Recommended fix: a `Deno.env.get("REMINDERS_SECRET")` header check + matching cron header. **Not deployed this pass** — requires `supabase functions deploy` against your live project, which I won't do without you asking for it explicitly.
- **`sw.js` `notificationclick` origin allow-list**: not independently exploitable — it's entirely downstream of the RLS item above. If that's fixed, this is moot without any change here.

### Confirmed clean, no action needed
- `import.meta.env` usage: only the 4 expected vars (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_VAPID_PUBLIC_KEY`, `DEV`/`PROD`) — no `service_role`, no LLM key baked into the build.
- AI key handling: user-entered at runtime, `localStorage`-only, by design (own comment in `ai.ts`).
- `npm audit --production`: 0 vulnerabilities.
- `agentation` dev tool: correctly gated behind `import.meta.env.DEV`, dynamic import, confirmed absent from the prod bundle (only an inert changelog *text* mention survives, not the module).

## Core UX flows (daily loop)

### Fixed
- **`src/lib/nlp.ts` — the flagship bug of this audit.** JavaScript's `\b` word boundary is defined against `[A-Za-z0-9_]` and never matches next to Cyrillic letters, so `/\bзавтра\b/i`, `/\bсегодня\b/i`, `/\bпослезавтра\b/i`, and every weekday-name regex silently never matched at all. Typing "написать завтра" into quick-add set **no due date** and left the literal word in the title — confirmed live via direct Node execution before the fix. Replaced with Unicode-aware lookaround boundaries (`(?<![\p{L}\p{N}_])word(?![\p{L}\p{N}_])`, `u` flag). Added `nlp.test.ts` (8 tests) so this can't regress silently again.
- **UTC-vs-local "today" bug, swept across the whole codebase.** `todayStr()`/`isToday()` (and ~15 other call sites independently reimplementing the same `toISOString().slice(0,10)` idiom) derived "today" in UTC, not local time — for this app's Moscow-timezone user, the calendar day flipped 3 hours early every evening, silently affecting streaks, overdue flags, completion-log bucketing, and time-section grouping. Added `localDayStr()`/`addDays()` to `format.ts` as the one canonical implementation, then swept every call site to use it: `DataProvider.tsx` (`streakOf`, `toggleTask`, `advanceDate`), `gameStats.ts`, `taskGrouping.ts`, `analytics.ts`, `aiContext.ts`, `ContributionGrid.tsx`, `GamificationEngine.tsx`, `seed.ts`, `Knowledge.tsx`, `Tasks.tsx` (`offsetDate`/`nextMonday`/`reschedule`), `ProjectDetail.tsx` and `TaskEditDialog.tsx`'s due-date quick-menus, `Dashboard.tsx` (streak/tomorrow/dynamics/weekByDay). Left `SettingsPage.tsx`'s export-filename timestamps alone (cosmetic only, no data impact). Added `format.test.ts` (6 tests) pinning behavior at both a late-evening and just-after-local-midnight instant.
- **Monthly recurrence day-overflow.** `advanceDate`'s bare `else` branch (`recurrence: "monthly"`) did a naive `setMonth(getMonth()+1)`, which overflows into the month *after* next when the source day doesn't exist there — e.g. a task due Jan 31 recurring monthly silently skipped February entirely, landing on Mar 3. Now clamps to the target month's actual last day. Added 4 new recurrence tests to `flows.test.tsx` (monthly-overflow, weekly, weekdays-skip-weekend, monthly-first-monday) — all pass.
- **Dashboard/ProjectDetail inline "add task" bypassed NLP parsing entirely** — typing `#клиент !важно завтра` created a task literally titled that way, no tags/importance/date extracted, while the same string in the global QuickAddDialog parsed correctly. Both inline adders now call `parseNaturalInput` and merge `tags`/`important`/`dueDate` with the same "explicit date-picker pick wins over the NLP guess" precedence QuickAddDialog uses (Dashboard needed a small `dateTouched` flag since its picker always has *some* value, unlike QuickAddDialog's empty-by-default override).
- **QuickAddDialog didn't seed `projectId` from the current route** — opening quick-add from inside `/projects/:id` always defaulted to "no project." `AppLayout.tsx` now reads the route via `useMatch("/projects/:id")` and passes `defaultProjectId` through; the dialog seeds and resets to it.
- **`pushRecent` was only ever called from `ProjectDetail.tsx`** — Command Palette's "Недавнее" group could never surface a recently-opened task or note. Added a shared `openTask()` wrapper in `Tasks.tsx` (used by the list-row click, kanban Enter key, kanban card click, context-menu "Редактировать," and the `openTaskId` deep-link effect) and a `pushRecent` call in `Notes.tsx`'s `openEdit`. Since tasks/notes share one base route (`/tasks`, `/notes`) instead of per-item URLs, `RecentItem` gained an optional `state` field and `pushRecent`'s dedupe key changed from `to` to `id` (deduping by `to` would have collapsed every recently-opened task into one slot).

### Deferred
- **Tasks.tsx multi-select has no visible affordance** — still Ctrl/Cmd/Shift+click only, discoverable via the header description text but no checkbox/select-all UI. This is a UI *addition*, not a bugfix, and moderate in scope (hover checkbox + "select all visible" control) — deferred to keep this pass to fixes rather than new interaction surface.

## Component architecture & duplication

### Confirmed, not fixed this pass (larger refactors, real regression risk if rushed)
- **`Tasks.tsx` is 1320 lines**, one `export default function Tasks()` with filter/sort/group logic, bulk-action handlers, list-row render, kanban-card render, kanban-column CRUD, and the right-click context menu all as nested closures — zero extracted sub-components.
- **The due-date `DropdownMenuContent` (today/tomorrow/+7d/clear) is duplicated 5 times** across `Tasks.tsx` (list row, kanban card, context menu — which is missing the +7d/clear items entirely, an existing behavioral drift) plus `ProjectDetail.tsx` and `TaskEditDialog.tsx`. This pass's UTC-fix sweep already routed every copy through the shared `addDays()` helper (removing the *date-arithmetic* duplication), but the markup/menu-shape duplication itself remains — extracting one `<DueDateMenu>` is the natural next step and is now easier since the arithmetic is already unified.
- **No shared `<TaskRow>`** — `Tasks.tsx`'s `renderRow`/`kanbanCard` and `ProjectDetail.tsx`'s `renderTaskRow` are three hand-separate reimplementations with confirmed drift in feature set (rename, tag-add, indicator icons present in list view only) — the delete-without-undo drift specifically **was** fixed this pass (see Data layer/UX section), but the row markup itself stays unconsolidated.
- **`DataProvider.tsx`'s context value is still one `useMemo` keyed on `[data]`** wrapping the data snapshot + all 60 action closures. Splitting into a stable "actions" context is possible (most actions are pure `setData` updaters) but 4 closures (`toggleTask`, `toggleMeeting`, `archiveDoneTasks`'s count, `clientRisk`) read the outer `data` directly and need individual triage first — not a mechanical one-pass split, deferred.
- **`React.memo`: still 0 usages** — not worth introducing before the row-extraction above gives it something stable to wrap.

**Why deferred as a group:** these four are one underlying root cause (no extraction boundary between state and presentation) and the fix is the *same* piece of work for all of them — extracting `TaskRow`/`DueDateMenu` and splitting the context. Doing it properly needs its own focused pass with careful before/after behavior checks (especially since `ProjectDetail`'s delete-undo bug this pass showed real drift can hide in these duplicates), not a rushed mechanical extraction bundled into a 20-item audit sweep.
